# web-pipeline

웹 제작 파이프라인을 4단계로 나눠 각 단계의 작업 지침을 담은 플러그인.
**PHP 백엔드 + JS 프론트 분리 구조** 기준.

## 담긴 스킬

| 스킬 | 단계 | 활성화 예시 |
|---|---|---|
| `web-planning` | 기획 | "요구사항 정리", "IA 짜줘", "화면 설계", "유저플로우" |
| `web-design` | 디자인 | "디자인 토큰", "컴포넌트 설계", "반응형 기준", "스타일 가이드" |
| `web-markup` | 마크업 | "마크업", "시맨틱 HTML", "퍼블리싱", "접근성 보완" |
| `web-frontend` | 프론트 | "API 연동", "fetch로 데이터", "상태 관리", "폼 검증" |

각 스킬은 해당 단계의 작업을 요청하면 자동으로 활성화된다.

## 단계 흐름

기획 → 디자인 → 마크업 → 프론트. 앞 단계의 산출물이 다음 단계의 입력이 된다.

- 기획: 요구사항·IA·화면정의·API 계약 초안
- 디자인: 디자인 토큰·컴포넌트 명세·반응형 규칙
- 마크업: 시맨틱 HTML·CSS 토큰 구현·접근성
- 프론트: JS로 PHP API 연동·상태·폼·로딩/에러 UI

## 핸드오프 기록 (핵심 기능)

구두·메신저 전달은 휘발된다. 그래서 각 단계는 끝날 때 **정해진 양식의 기록을 `docs/`에 남기고**, 다음 단계는 시작할 때 그 기록을 먼저 읽는다. 회의에 없던 사람도 파일만 읽으면 따라잡을 수 있다.

| 단계 | 남기는 기록 | 읽는 기록 |
|---|---|---|
| 기획 | `docs/01-planning.md` | — |
| 디자인 | `docs/02-design.md` | `01-planning.md` |
| 마크업 | `docs/03-markup.md` | `02-design.md` |
| 프론트 | `docs/04-frontend.md` | `03-markup.md`, `01-planning.md` |

기본은 작업 폴더의 마크다운이며, 노션 등 협업 도구에 그대로 붙여넣어 공유할 수 있다.

## 구조 가정

- PHP는 REST API만 제공 (화면을 그리지 않음)
- 화면은 브라우저 JS가 API를 호출해 렌더
- CSS 방식은 프로젝트에 맞게 (토큰은 CSS 변수 기준)
