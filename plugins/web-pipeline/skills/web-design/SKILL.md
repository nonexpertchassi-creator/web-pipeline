# 웹 디자인 (Design)

기획에서 정한 화면을 시각·구조 규칙으로 옮기는 단계. 산출물(토큰·컴포넌트 명세)은 마크업/프론트가 그대로 구현할 수 있어야 한다.

## 시작 전: 기획 기록 읽기

작업 폴더에 `docs/01-planning.md`가 있으면 **먼저 읽는다.** 거기 담긴 화면 정의·요구사항·API 계약이 디자인의 전제다. 없으면 사용자에게 기획 기록이 있는지 묻고, 없으면 web-planning을 먼저 권한다.

## 시작 전 확인

- 기획 산출물(화면 목록·컴포넌트 후보)이 있는가.
- 기존 디자인 토큰/브랜드 가이드가 있는가. 있으면 그것을 따른다.
- CSS 구현 방식 제약이 있는가. (없으면 토큰을 CSS 변수로 정의해 어떤 방식에도 얹히게 한다)

## 작업 순서

1. **디자인 토큰 정의** — 원자 값을 먼저 정한다. 하드코딩 색/사이즈를 코드에 흩지 않기 위함.
   - 색: 의미 기반으로 (`--color-text`, `--color-bg`, `--color-primary`, `--color-danger` …). 라이트/다크 둘 다 고려.
   - 타이포: 폰트 패밀리, 크기 스케일, 굵기, 행간.
   - 간격: 4·8px 배수 같은 일관된 스케일.
   - 반경·그림자·border 등.
2. **컴포넌트 설계** — 화면을 재사용 단위로 쪼갠다. 버튼·인풋·카드·모달 등. 각 컴포넌트의 상태(기본/호버/포커스/비활성/에러)를 명세한다.
3. **레이아웃·반응형** — 브레이크포인트와 그리드 규칙을 정한다. 모바일 우선으로 작은 화면부터 정의한다.
4. **상태 디자인** — 빈 상태·로딩·에러 화면을 반드시 디자인한다. 기획에서 정의한 분기와 1:1 대응.

## 산출물 형식

- 토큰: CSS 변수 목록 또는 표 (이름 · 값 · 용도)
- 컴포넌트: 컴포넌트별 명세 (역할 · props/변형 · 상태별 모양)
- 반응형: 브레이크포인트 표 + 화면별 동작

토큰은 가능하면 실제 CSS 변수 블록으로 떨어뜨려 마크업 단계가 복붙할 수 있게 한다.

## 원칙

- **토큰 먼저, 화면 나중.** 색/간격을 컴포넌트마다 새로 정하면 일관성이 깨진다.
- **의미로 이름 짓는다.** `--blue-500`보다 `--color-primary`. 나중에 색이 바뀌어도 이름은 유지된다.
- **접근성을 디자인 단계에서 챙긴다.** 본문 텍스트 대비비 4.5:1 이상, 포커스 표시를 디자인에 포함, 색만으로 정보를 전달하지 않기.
- **컴포넌트는 상태가 핵심이다.** 기본 모양만 그리고 호버·에러·비활성을 빼면 마크업에서 즉흥적으로 채우게 된다.
