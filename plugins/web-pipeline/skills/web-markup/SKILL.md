# 웹 마크업 (Markup)

디자인을 실제 HTML 구조와 CSS로 옮기는 단계. 분리 구조에서는 PHP가 화면을 그리지 않으므로, 마크업은 JS 프론트의 컴포넌트 템플릿 또는 정적 HTML 안에 들어간다. 이 단계의 산출물은 **데이터·이벤트가 붙기 직전의, 구조와 표현이 완성된 빈 화면**이다.

이 스킬은 **단독으로 쓸 수 있다.** 앞 단계(기획·디자인) 기록이 없어도 시작하며, 있으면 그것을 기준으로 삼는다.

## 시작 전: 입력 확인 (있으면 따르고, 없어도 시작)

마크업은 디자인 기록이 없어도 시작할 수 있다. 다만 무엇을 기준으로 그릴지는 정하고 들어간다.

1. **디자인 기록이 있으면** — 작업 폴더의 `docs/02-design.md`나 피그마 링크에서 토큰·컴포넌트 명세·반응형 규칙을 가져와 그대로 따른다. 새로 발명하지 않는다.
2. **없으면** — 임의로 디자인 체계를 깔지 말고, **이 화면 범위에서 필요한 것만** 사용자와 최소로 합의한다:
   - 어떤 색·간격·폰트를 쓸지 (토큰화할지는 아래 "토큰을 CSS 변수로" 참조)
   - 브레이크포인트 기준값 (없으면 콘텐츠가 깨지는 지점에서 잡는다)
   - 마크업이 들어갈 곳: JS 프레임워크 컴포넌트인가, 정적 HTML인가
3. **큰 그림이 없어 막막하면** — 디자인 체계를 새로 세우는 건 마크업 단계의 일이 아니라고 짚어주고, 필요하면 디자인 단계를 권한다. 단, 그 때문에 이번 화면을 멈추지는 않는다. 이 화면에 필요한 만큼만 정하고 진행한다.

## 시작 전 확인

- 마크업이 들어갈 곳: JS 프레임워크 컴포넌트인가, 정적 HTML인가. (작명·파일 분리 방식이 달라진다)
- 토큰/컴포넌트 명세가 있는가. 없으면 위 "입력 확인"대로 이 화면 범위만 합의.
- CSS 방식 제약이 있는가. 없으면 **토큰은 CSS 변수, 클래스는 작명 규칙 하나(BEM 등)** 로 통일.
- 브레이크포인트 기준값이 정해져 있는가. 없으면 콘텐츠 기준으로 잡는다.

## 시작 전: 프로젝트 컨텍스트 → 렌더링·메타 소유 (먼저 — 위험·낭비 방지)

**0. 프로젝트 컨텍스트(`CLAUDE.md`)부터 읽는다.** 프로젝트 루트의 `CLAUDE.md`에 렌더링 방식·`<head>`/메타 소유·토큰 위치·브레이크포인트 같은 **고정 사실**이 적혀 있으면 그걸 신뢰하고 **재조사하지 않는다.** 이 사실들은 화면마다 바뀌지 않으므로, 세션마다 코드를 뒤지거나 사용자에게 다시 묻는 건 토큰 낭비이자 답이 흔들리는 원인이다. (`CLAUDE.md`는 Claude가 보통 세션마다 자동으로 읽는다.)

**새 페이지라고 `<html>`부터 만들지 않는다.** 실무 페이지 대부분은 독립 문서가 아니라 공유 레이아웃에 끼는 *콘텐츠 조각*이고, `<head>`·메타는 다른 데서 소유한다. 아래 두 가지를 확인하되, **`CLAUDE.md`에 이미 적혀 있으면 그걸 따르고, 없을 때만 실제로 파악한다.**

1. **이 작업 단위는 무엇인가**
   - **콘텐츠 조각** — 공유 레이아웃(헤더·head·푸터 include, 템플릿 엔진, JS 컴포넌트)에 들어간다. → `<html>`/`<head>`를 만들지 않는다. `<main>` 안 콘텐츠부터 작성.
   - **독립 문서** — 정말 단독 HTML 파일이다(드묾). → 이때만 `<html><head>`를 직접 만든다.

2. **`<head>`·메타는 누가 소유하나**
   - **외부 SEO 메타 관리자** — title·description·OG 등이 관리 화면에서 주입된다. → 페이지에 메타 태그를 쓰지 말고 그 도구에 값을 넣는다. 페이지에 또 박으면 중복·충돌.
   - **공유 head include + 변수** — `<head>`를 include하고 페이지는 변수만 채운다(예: PHP `$pageTitle`, `$metaDescription`). → 태그가 아니라 *그 변수*를 세팅한다.
   - **페이지 자신이 소유** — 그제서야 `<head>`에 직접 title·meta를 쓴다.

**파악했으면 `CLAUDE.md`에 적어둔다.** `CLAUDE.md`에 위 고정 사실이 없으면, 이번에 알아낸 걸 한 줄씩 적어 다음 작업자·다음 단계가 재조사하지 않게 한다. 적을 것: 렌더링 방식(예: PHP API + JS 렌더/CSR), `<head>`·메타 소유(예: 외부 SEO 메타 관리자 + 공유 head include `$변수`), 토큰 정의 위치, 브레이크포인트 기준값, 클래스 작명 규칙. (`CLAUDE.md`가 없으면 만들어도 된다 — 사이트 프로젝트 루트에.)

> 요약: 고정 사실은 `CLAUDE.md`에서 **읽고(없으면 한 번 적고)**, `<head>`·메타(title·description·canonical·OG·charset·viewport·lang)는 기본적으로 "바깥 소유"라고 가정한다. 마크업의 본 산출물은 거의 항상 `<head>`가 아니라 `<body>`/`<main>` 안 구조다.

## 작업 순서

1. **문서 골격부터** — 의미 있는 랜드마크로 페이지 뼈대를 먼저 잡는다. `header` / `nav` / `main`(페이지당 하나) / `section` / `article` / `aside` / `footer`. `div`·`span`은 의미 태그로 표현이 안 될 때만. (위 "렌더링·메타 소유 파악"대로 — 콘텐츠 조각이면 `<html>`/`<head>`는 만들지 말고 `<main>` 안 구조부터.)
2. **토큰부터 확인** — 토큰이 변수로 정리돼 있으면 그걸 참조하고, 없으면 만들지·어디까지 만들지 사용자와 정한다(아래 "토큰을 CSS 변수로" 참조). 임의로 토큰 체계를 깔지 않는다.
3. **컴포넌트 단위로 마크업** — 디자인에서 쪼갠 컴포넌트대로 HTML 조각을 만든다. 같은 컴포넌트는 같은 구조·같은 클래스로. 복붙 후 한 군데만 고치는 일이 없게 한다.
4. **상태를 클래스/속성으로 미리 표현** — 에러·비활성·열림 등 상태를 클래스(`.is-error`)나 표준 속성(`[disabled]`, `[aria-expanded]`)으로 마크업에 심어두고 CSS로 그린다. 프론트는 이 클래스·속성만 토글하면 된다.
5. **반응형** — 모바일 우선으로 기본 스타일을 짜고 `min-width` 미디어쿼리로 키운다. 레이아웃은 flex/grid로.
6. **이미지 최적화** — 가져온 이미지가 있으면 webp 변환(1MB↑)·SEO 파일명·설명형 alt를 처리한다. 퍼블리셔가 자주 누락하는 지점. (아래 "이미지 최적화" / `reference/image-optimization.md`)
7. **접근성 점검 (페이지에 맞게)** — 마무리 전 `reference/a11y-checklist.md`를 본다. *전부 완수*가 목표가 아니라, 이 페이지의 목적·노출·이해관계에 맞게 어디까지 챙길지 판단한다. '항상' 항목(거의 공짜)은 늘 지키고, '판단' 항목은 페이지 성격에 따라 정한다.

## 시맨틱 태그 빠른 참조

의미 태그를 고를 때의 기본값. 헷갈리면 "이 요소를 스크린리더가 뭐라고 읽어야 하나"로 판단한다.

- **누르면 화면 안에서 뭔가 일어남** → `<button type="button">` (제출·토글·열기). `<div onclick>` 금지.
- **누르면 다른 곳으로 이동** → `<a href>`. 이동이 아니면 링크가 아니다. `href` 없는 `<a>`는 키보드로 못 닿는다.
- **목록** → `<ul>`/`<ol>` + `<li>`. 카드 그리드·메뉴·태그·탭도 의미상 목록이면 목록으로.
- **제목 위계** → `<h1>`~`<h6>`을 건너뛰지 않고 순서대로. 페이지당 `<h1>`은 하나. 크기 때문에 레벨을 바꾸지 말고 CSS로 키운다.
- **입력 폼** → `<form>` + `<label>` + `<input>/<select>/<textarea>`. 관련 입력 묶음은 `<fieldset>`+`<legend>`.
- **여닫는 패널·아코디언** → `<details>`/`<summary>` 우선 검토(공짜로 동작·접근성 제공).
- **표 형식 데이터** → `<table>` + `<thead>`/`<tbody>` + `<th scope>`. 레이아웃 용도로는 절대 쓰지 않는다.
- **그림·도표 + 설명** → `<figure>` + `<figcaption>`.
- **탐색 묶음** → `<nav>` (주 메뉴·푸터 메뉴·브레드크럼). 여러 개면 `aria-label`로 구분.
- **시간·날짜** → `<time datetime="…">`. 기계가 읽을 값과 보이는 텍스트를 분리.

## 토큰을 CSS 변수로 (먼저 확인부터)

토큰이 CSS 변수로 정리돼 있는 환경은 오히려 드물다. **있다고 전제하지 말고 먼저 확인한다.** (디자인 단계를 거쳤다면 `docs/02-design.md`에 토큰 블록이 있을 가능성이 높으니 거기부터 본다.)

1. **확인** — 디자인 기록(`02-design.md`)이나 코드베이스에 토큰 정의(`:root` 변수, 스타일 가이드, 테마 파일 등)가 이미 있는가.
2. **있으면** — 그걸 그대로 참조한다. 새로 만들지 않는다. 작명·구조를 따라간다.
3. **없으면 — 만들지부터 물어본다.** 임의로 토큰 체계를 깔지 않는다. 사용자에게 묻는다:
   - "지금 색·간격·폰트가 토큰으로 정리돼 있지 않은데, 이번에 CSS 변수로 만들까요? 아니면 이 화면 범위에서만 값을 직접 쓸까요?"
4. **범위를 합의한다** — 만들기로 했다면 어디까지인지 정한다:
   - **이 화면만** — 이 컴포넌트에서 반복되는 값만 지역 변수로. 전역 체계는 손대지 않는다.
   - **전역 토큰** — `:root`에 프로젝트 공통 토큰을 깐다. 디자인 단계(web-design)의 일이 밀려온 거라면, 토큰 정의 자체는 디자인 기록에 남기는 게 맞다고 짚어준다.
   - 둘 다 아니면 — 토큰화 없이 값 직접 사용도 선택지다. 일관성만 클래스로 관리.

> 핵심: 마크업 단계가 **임의로 토큰 체계를 발명하지 않는다.** 있으면 따르고, 없으면 만들지·어디까지 만들지를 사용자와 정한 뒤 움직인다.

토큰을 만들기로 합의했을 때의 형태:

```css
:root {
  --color-text: #1a1a1a;
  --color-danger: #d33;
  --space-2: 8px;
  --space-3: 16px;
  --font-size-body: 16px;
  --radius: 8px;
}
.btn { padding: var(--space-2) var(--space-3); border-radius: var(--radius); }
.field--error { color: var(--color-danger); }
```

- 토큰을 쓰기로 했다면, 값을 직접 박고 싶어질 때마다 "이건 토큰인가"를 묻는다. 색·간격·폰트·반경·그림자는 거의 항상 토큰이다.
- 다크모드·테마가 있으면 변수 값만 갈아끼우면 되도록 컴포넌트에 색을 직접 박지 않는다.

## 폼·인터랙티브 패턴

분리 구조에서 폼은 프론트가 가로채 API로 보낸다. 마크업은 **그 직전까지의 올바른 구조**를 만든다.

- 모든 입력에 `<label for>` 또는 `<label>` 감싸기. placeholder는 라벨 대체가 아니다(포커스하면 사라지고 스크린리더가 라벨로 읽지 않는다).
- 입력 타입을 정확히: `type="email"`, `type="tel"`, `type="number"`, `type="date"`, `type="url"`, `type="search"`. 브라우저 검증·모바일 키보드가 공짜로 따라온다.
- 자동완성 의도를 `autocomplete`로 명시(`name`, `email`, `current-password` 등). 사용자 입력 부담이 준다.
- 필수값은 `required`, 형식 제약은 `pattern`/`min`/`max`/`minlength`로 1차 표현. (최종 검증은 서버 몫이지만 마크업이 의도를 드러낸다)
- 에러 메시지 자리를 미리 만들고 입력과 `aria-describedby`로 연결. 에러 상태는 `aria-invalid="true"`로 표시.
- 제출 버튼은 `<button type="submit">`. 폼 안의 일반 버튼은 `type="button"`을 **명시**(기본값이 submit이라 의도치 않은 제출이 생긴다).
- 토글 버튼은 `aria-pressed`, 펼침 버튼은 `aria-expanded`로 상태를 노출한다. 보이는 모양만 바꾸지 말 것.
- 입력 그룹(라디오·체크박스 묶음)은 `<fieldset>`+`<legend>`로 묶어 무엇에 대한 선택인지 들리게 한다.

## 반응형

- **모바일 우선**: 기본 스타일 = 가장 좁은 화면. `@media (min-width: …)`로 넓힐 때만 덮어쓴다.
- 브레이크포인트는 디자인 기록 값을 쓰고, 없으면 콘텐츠가 깨지는 지점에서 잡는다(기기 모델명 기준 X).
- 고정 px 폭을 남발하지 않는다. `max-width` + `width: 100%`, `flex`/`grid`로 흐르게.
- 이미지·미디어는 `max-width: 100%; height: auto`를 기본으로. 이미지가 늦게 떠도 콘텐츠가 안 밀리게 **자리를 미리 잡는다**(레이아웃 흔들림=CLS 방지) — `width`/`height` 속성이든 CSS `aspect-ratio`든 고정 크기 컨테이너든 프로젝트 방식대로. (다운로드 속도가 아니라 레이아웃 안정 얘기)
- 폰트·간격에 `rem` 우선. 사용자가 키운 글자 크기를 존중한다.
- 터치 대상은 최소 44×44px 확보(좁은 화면에서 누르기 쉽게).

## 이미지 최적화 (webp · 파일명 · alt)

피그마 등에서 가져온 이미지는 그대로 넣지 말고 처리한다. 전체 절차·예시는 `reference/image-optimization.md`.

- **webp 변환** — 1MB 넘는 png/jpg는 webp로(품질 ~80). `bash ${CLAUDE_SKILL_DIR}/scripts/optimize-images.sh <폴더>` (있는 도구로 변환, 원본 보존). 최신 브라우저 전부 지원.
- **SEO 파일명** — `IMG_2931.png` 대신 브랜드+내용 기반 이름. Claude가 이미지를 직접 보고 제안하되 브랜드·제품명은 `CLAUDE.md` 용어 참고(모르면 묻는다). 바꿀 땐 `bash ${CLAUDE_SKILL_DIR}/scripts/rename-image.sh <기존> <새이름>`으로 **참조까지 자동 갱신**. ⚠ CMS·DB·SEO 관리자 등 코드 밖 참조는 직접 확인.
- **설명형 alt** — 키워드 나열이 아니라 *보이지 않는 사람에게 설명하듯*. 예: `한맥 엑스트라 캔에 황금빛 'smoooth' 슬로건이 적혀 있다`. 장식용은 `alt=""`.

> 스크립트는 **셸이 있는 환경에서 Claude가 실행**한다(자동 실행 아님). 경로는 반드시 `${CLAUDE_SKILL_DIR}/...`로 — 사용자 현재 폴더 기준 상대경로는 안 맞는다. **필요 도구:** cwebp·ImageMagick·Pillow 중 하나(없으면 스크립트가 알려준다).

## 접근성 (비례적으로 — 전체는 reference 파일)

접근성은 체크리스트 전부 완수가 아니라 **이 페이지에 얼마나 필요한지 판단하는 일**이다(SEO와 같은 성격). 아래 '항상' 4가지는 거의 공짜라 — 사실 그냥 올바른 마크업이라 — 페이지 종류와 무관하게 지킨다. 그 이상은 페이지의 노출·수익·규제 정도에 맞춰 `reference/a11y-checklist.md`의 '판단' 항목에서 고른다.

**항상 (거의 공짜):**

- 인터랙티브 요소는 `div`가 아니라 `button`/`a`로 — 키보드 포커스·엔터가 공짜로 따라온다.
- 모든 이미지에 의미 있는 `alt` (장식용은 `alt=""`), 모든 폼 요소에 연결된 `label`.
- 제목 위계(`h1`~`h6`)를 건너뛰지 않는다. 색만으로 정보를 전달하지 않는다.
- 포커스 표시(`:focus-visible`)를 지우지 않는다.

**판단 (페이지 성격에 따라):** 스크린리더 실제 테스트, 200% 확대 검증, skip link, 정밀 대비 측정, `aria-live` 등. 공개·수익·규제에 가까울수록 끌어올리고, 내부용·단명 페이지면 항상 항목만 단단히 하고 멈춰도 된다. 자세한 선 긋기 기준은 reference 파일 참조.

## 검색 노출 (SEO — 비례적으로)

SEO도 a11y와 같은 성격이다 — 고정 목록 완수가 아니라 **이 페이지가 검색·외부 유입 대상인지**에 맞춰 투자한다. 마크업은 그중 **온페이지/구조적 SEO**만 소유한다. 전체 목록·선 긋기·핸드오프는 `reference/seo-checklist.md` 참조.

**항상 (거의 공짜 — a11y와 겹친다):** 페이지마다 고유한 `<title>`·`<meta description>`, 제목 위계, 시맨틱 구조, 이미지 `alt`, 진짜 `<a href>`(JS onclick 이동 X), `lang`·`viewport` 메타. 검색을 안 노려도 해두면 손해가 없다. (단, `<head>` 항목인 title·description·lang·viewport는 위 "렌더링·메타 소유 파악"대로 — 바깥(SEO 관리자·공유 head 변수)이 소유하면 직접 쓰지 말고 거기에 값만 넣는다.)

메타 값(title·description·keywords)은 **비워두지 말고** 페이지 콘텐츠 기반 실제 값으로 채운다 — 더미라도 lorem ipsum 말고 내용 기반으로. 소유 위치(SEO 관리자·`$변수`·페이지)에 넣는다. (자세히는 `reference/seo-checklist.md`)

**Open Graph·Twitter Card도 팀 필수**라 항상 넣는다(공유 카드 미리보기 — `og:title/description/image/url/type`, `twitter:card`). 값은 title·description과 같은 소스에서, 소유 위치 규칙대로.

**판단 (페이지 성격에 따라):** 구조화 데이터(JSON-LD), `canonical`, `robots`, `hreflang` 등. 공개·검색이 가치 있을수록 챙기고, 로그인 뒤 내부 화면이면 대부분 건너뛴다(`noindex`가 맞기도).

> ⚠ **이 프로젝트의 핵심 SEO 리스크는 마크업이 아니라 렌더링이다.** PHP가 API만 주고 화면을 JS가 그리면(CSR), 크롤러가 본문을 못 볼 수 있다. 마크업을 완벽히 해도 검색에 안 잡힌다. 이 확인은 **프론트/아키텍처 몫**이라 핸드오프로 넘긴다.

## 흔한 안티패턴 (피한다)

- `<div onclick>` 버튼 — 키보드·포커스·스크린리더가 전부 빠진다. `<button>`을 써라.
- `<table>`로 레이아웃 짜기 / `<br>`로 간격 주기 — 간격은 CSS.
- 제목 크기 때문에 `<h3>`을 `<h1>`로 바꾸기 — 위계는 의미, 크기는 CSS.
- 토큰 무시하고 색·간격 하드코딩 — 테마·일관성이 무너진다.
- placeholder를 라벨 대신 쓰기 — 포커스하면 사라지고 접근성이 깨진다.
- ARIA 떡칠 — 잘못된 ARIA는 없느니만 못하다. 시맨틱 태그가 먼저.
- 복붙한 컴포넌트를 한 군데만 수정 — 같은 컴포넌트는 같은 구조로 유지.

## 원칙

- **네이티브 우선.** 날짜는 `<input type="date">`, 토글은 `<button aria-pressed>`, 여닫기는 `<details>`. 브라우저가 이미 주는 걸 다시 만들지 않는다.
- **시맨틱이 곧 접근성이자 SEO다.** 올바른 태그를 쓰면 스크린리더·검색엔진이 공짜로 이해한다.
- **클래스는 일관되게.** 작명 규칙(BEM 등) 하나를 정해 끝까지 지킨다.
- **마크업에 로직을 섞지 않는다.** 데이터 바인딩·이벤트는 프론트 단계의 몫. 마크업은 구조와 표현까지, 그리고 동작이 붙을 자리(클래스·속성)를 표시하는 데서 멈춘다.

## 핸드오프 기록 (권장)

마크업이 끝나면 기록을 남긴다. 단독으로 작업했더라도, 나중에 동작을 붙이는 사람(자신이든 남이든)이 마크업을 다시 해석하지 않게 해준다.

- 저장 위치: `docs/03-markup.md`
- 양식:
  - 만든 마크업 파일/컴포넌트 목록
  - 클래스 작명 규칙
  - **동적 영역 표시** — "여기에 데이터가 들어간다", "이 버튼에 동작이 붙는다", "이 클래스를 토글해 상태를 바꾼다"를 구체적으로
  - 정의한 CSS 변수(토큰) 위치
  - **접근성 범위 결정** — 이 페이지에서 챙긴 것 / 의도적으로 건너뛴 것 + 이유(페이지 성격). '남은 결함'이 아니라 '정한 범위'로 적는다. 프론트로 넘기는 동작성 항목(포커스 트랩·라이브 영역)도 여기에.
  - **SEO 범위 결정** — 챙긴 온페이지 요소 / 건너뛴 것 + 이유. 특히 **크롤링 가능성(CSR) 확인을 프론트로 넘긴다**고 명시. (자세히는 `reference/seo-checklist.md`)
- 목표는 다음 작업자가 마크업을 다시 해석하지 않고 **데이터·이벤트만 꽂으면 되도록** 만드는 것.
