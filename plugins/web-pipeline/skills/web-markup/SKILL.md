# 웹 마크업 (Markup)

디자인을 실제 HTML 구조와 CSS로 옮기는 단계. 분리 구조에서는 PHP가 화면을 그리지 않으므로, 마크업은 JS 프론트의 컴포넌트 템플릿 또는 정적 HTML 안에 들어간다.

## 시작 전: 디자인 기록 읽기

작업 폴더에 `docs/02-design.md`가 있으면 **먼저 읽는다.** 토큰·컴포넌트 명세·반응형 규칙이 마크업의 기준이다. 없으면 사용자에게 디자인 기록(또는 피그마 링크)이 있는지 묻는다.

## 시작 전 확인

- 디자인 토큰/컴포넌트 명세가 있는가.
- 마크업이 들어갈 곳: JS 프레임워크 컴포넌트인가, 정적 HTML인가.
- CSS 방식 제약이 있는가. (없으면 토큰 CSS 변수 + 평범한 클래스로)

## 작업 순서

1. **시맨틱 구조 먼저** — `div` 떡칠 전에 의미 태그로 뼈대를 잡는다. `header`/`nav`/`main`/`section`/`article`/`aside`/`footer`, 목록은 `ul`/`ol`, 버튼은 `button`, 링크는 `a`.
2. **토큰을 CSS 변수로** — 디자인 토큰을 `:root`에 변수로 깐 뒤 그 변수만 참조한다. 색·간격을 매번 하드코딩하지 않는다.
3. **컴포넌트 단위로 마크업** — 디자인에서 쪼갠 컴포넌트대로 HTML 조각을 만든다. 상태별 클래스(`.is-error`, `[disabled]`, `[aria-expanded]`)를 디자인 명세대로 반영.
4. **반응형** — 모바일 우선으로 기본 스타일을 짜고 `min-width` 미디어쿼리로 키운다. 레이아웃은 fl/grid로.
5. **접근성 점검** — 아래 체크리스트를 돌린다.

## 접근성 체크리스트 (생략 금지)

- 모든 이미지에 의미 있는 `alt` (장식용은 `alt=""`).
- 폼 요소마다 `label`이 연결돼 있다.
- 인터랙티브 요소는 `div`가 아니라 `button`/`a`로 — 키보드 포커스·엔터가 공짜로 따라온다.
- 포커스 표시(`:focus-visible`)를 지우지 않는다.
- 색만으로 정보를 전달하지 않는다 (에러를 빨강만으로 X, 텍스트/아이콘 병행).
- 본문 대비비 4.5:1 이상.
- 동적 영역엔 적절한 ARIA (`aria-live`, `aria-expanded`, `role`)를 최소한으로.

## 원칙

- **네이티브 우선.** 날짜는 `<input type="date">`, 토글은 `<button aria-pressed>`. 브라우저가 이미 주는 걸 다시 만들지 않는다.
- **시맨틱이 곧 접근성이자 SEO다.** 올바른 태그를 쓰면 스크린리더·검색엔진이 공짜로 이해한다.
- **클래스는 일관되게.** 작명 규칙(BEM 등) 하나를 정해 끝까지 지킨다.
- **마크업에 로직을 섞지 않는다.** 데이터 바인딩·이벤트는 프론트 단계의 몫. 마크업은 구조와 표현까지.

## 핸드오프 기록 (필수)

마크업이 끝나면 기록을 남긴다.

- 저장 위치: `docs/03-markup.md`
- 양식: 만든 마크업 파일/컴포넌트 목록 · 클래스 작명 규칙 · 동적으로 바뀌는 영역(프론트가 데이터·이벤트를 붙일 지점) 표시 · 접근성 체크 결과.
- 특히 "여기에 데이터가 들어간다", "이 버튼에 동작이 붙는다"를 명시해 프론트가 헤매지 않게 한다.
