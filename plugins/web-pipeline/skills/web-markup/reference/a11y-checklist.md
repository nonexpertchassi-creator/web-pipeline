# 접근성 — 비례적 판단 가이드

**접근성은 이 목록을 전부 통과하는 게 목표가 아니다.** 이 페이지의 목적·노출·이해관계에 맞게 *어디까지* 챙길지를 판단하는 일이다. (SEO와 같은 성격 — 고정된 완수가 아니라 페이지에 맞는 투자.)

핵심은 두 층으로 나뉜다:

- **항상** — 거의 공짜다. 처음에 제대로 쓰면 비용이 0이고, 안 하면 *모든* 사용자에게 손해이고 마크업도 지저분해진다. 페이지 종류와 무관하게 지킨다.
- **판단** — 비용이 든다(테스트·추가 마크업·검증 시간). 페이지가 공개적·수익적·규제 대상일수록 더 챙기고, 내부용·단명 페이지면 의식적으로 건너뛴다. 건너뛴 건 결함이 아니라 결정이다.

---

## 항상 (거의 공짜 — 페이지 무관하게 지킨다)

이 항목들은 "접근성"이라기보다 그냥 **올바른 마크업**이다. 지키는 데 추가 비용이 거의 없다.

- [ ] 인터랙티브 요소는 `div`/`span`이 아니라 `button`/`a`/네이티브 폼 요소다. (`<div onclick>` 금지 — 키보드·포커스가 공짜로 따라온다)
- [ ] 누르면 이동=`<a href>`, 같은 화면에서 동작=`<button>`. 역할이 섞이지 않았다.
- [ ] 모든 폼 입력에 연결된 `<label>`이 있다. placeholder로 대체하지 않았다.
- [ ] 입력 타입이 용도에 맞다(`email`/`tel`/`number`/`date` 등).
- [ ] 의미 있는 이미지에 내용을 설명하는 `alt`, 장식용은 `alt=""`.
- [ ] 제목 위계(`h1`~`h6`)가 순서대로이고 레벨을 건너뛰지 않는다. 페이지당 `h1`은 하나.
- [ ] 랜드마크(`header`/`nav`/`main`/`footer`)로 페이지 골격이 잡혀 있다.
- [ ] 포커스 표시를 지우지 않았다(`outline: none`만 덩그러니 두지 않기 — 필요하면 `:focus-visible`로 다시 그린다).
- [ ] 색만으로 정보를 전달하지 않는다(에러를 빨강만으로 X — 텍스트/아이콘 병행).
- [ ] 페이지에 `lang` 속성이 있다. (단 `<html lang>`은 head/레이아웃 소유 — 공유 레이아웃이 있으면 거기서 채워졌는지 확인, 페이지에 중복하지 않기)

> 위 항목을 어기는 건 "접근성을 덜 챙긴 것"이 아니라 "마크업을 틀리게 한 것"에 가깝다. 그래서 비용 대비 거의 항상 옳다.

## 판단 (페이지 성격에 따라 — 들이는 만큼 비용)

아래는 가치가 분명하지만 **시간·테스트·추가 마크업 비용**이 든다. 이 페이지에 그만큼의 투자가 맞는지 정한다.

- [ ] 본문 대비비 4.5:1 / 큰 텍스트 3:1 / UI 경계 3:1 — *정밀 측정.* (값 자체는 디자인 소유. 마크업은 디자인 토큰을 그대로 쓰면 자동 충족되는 경우가 많다)
- [ ] 키보드만으로 핵심 흐름(이동·입력·제출·열고닫기)을 끝까지 해본다. — 인터랙션이 많은 페이지일수록 필수에 가깝다.
- [ ] Tab 순서가 시각 순서와 일치한다(양수 `tabindex` 남용 X).
- [ ] 아이콘만 있는 버튼/링크에 접근 가능한 이름(`aria-label` 등).
- [ ] 에러 메시지를 `aria-describedby`로 입력과 연결, `aria-invalid="true"`로 표시. — 폼이 중요한 페이지(가입·결제·문의)일수록 챙긴다.
- [ ] 200% 확대 / 모바일 폭에서 잘리거나 가로 스크롤로 깨지지 않는다.
- [ ] 긴 내비 위에 "본문 바로가기"(skip link). — 내비가 길고 반복되는 공개 페이지에서 가치.
- [ ] 비동기 알림 영역에 `aria-live`(보통 `polite`). — *동작은 프론트 몫*, 마크업은 자리만 표시.
- [ ] 터치 대상 최소 44×44px. — 모바일 비중·연령대 따라.
- [ ] 스크린리더로 주요 화면을 실제로 읽혀본다. — 공개·수익·규제 페이지의 최종 검증.

> "판단" 항목 중에도 **모달 포커스 가두기·Esc 닫기·라이브 영역 갱신**처럼 *동작*에 속하는 건 프론트 단계 몫이다. 마크업은 자리(클래스·속성)만 표시하고 넘어간다.

## 선 긋기 (어디까지 할지 정하는 법)

다음을 자문해 "판단" 항목을 어디까지 끌어올릴지 정한다.

- **누가, 어떤 맥락에서 쓰나** — 불특정 공개 vs 내부 직원. (내부라도 장애가 있는 동료를 배제하진 않는지 한 번 생각)
- **이 페이지가 KPI·수익·전환과 얼마나 얽혀 있나** — 랜딩·결제·가입이면 끌어올린다. 임시·실험·단명 페이지면 항상 항목 + 최소만.
- **수명** — 오래 유지될 화면일수록 투자가 회수된다.

공개적·수익적·규제 대상에 가까울수록 "판단" 항목이 "필수"로 올라온다. 반대면 항상 항목만 단단히 하고 멈춰도 된다.

> **법적 최소선(참고):** 공공기관·일부 계약·특정 지역 규제에선 접근성이 *의무*다. 그런 맥락이면 위 "판단"이 사실상 "필수"가 되고 재량이 거의 없다. 해당 여부가 불확실하면 사용자에게 확인한다.

## 핸드오프 — 결함이 아니라 "범위 결정"으로 적는다

마무리하면 `docs/03-markup.md`에 **남은 결함 목록이 아니라 이 페이지의 접근성 범위 결정**을 남긴다:

- 챙긴 것 (어느 수준까지)
- **의도적으로 건너뛴 것 + 이유** (페이지 성격상 불필요 / 비용 대비 안 맞음 / 다른 단계 몫)
- 프론트로 넘기는 동작성 항목 (포커스 트랩·라이브 영역 등)

이렇게 적으면 다음 작업자가 "빠뜨린 것"과 "정한 것"을 구분할 수 있다.
