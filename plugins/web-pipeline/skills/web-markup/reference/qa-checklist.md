# 마크업 QA (핸드오프 전 self-QA)

**누가 하나:** "만든 사람이 핸드오프 전 **1차**, 기계가 **자동으로**, 사람이 **2차로**." 마크업 스킬이 담당하는 건 그중 **마크업 self-QA(1차) + 자동 점검**이다. 완전한 기능·E2E·실기기 QA는 프론트 단계나 전담 QA의 몫이라 그쪽으로 넘긴다.

**깊이는 페이지 성격에 비례**한다(a11y·SEO와 같은 원칙). 내부용·단명 페이지면 자동 점검 + 아래 핵심만. 공개·수익·규제 페이지면 사람 2차까지.

## 1. 자동 점검 먼저 (기계가 잡는 것)

```bash
bash ${CLAUDE_SKILL_DIR}/scripts/qa-check.sh <대상폴더>
```

판단 없이 잡히는 것들: **1MB 초과 미변환 이미지 / `alt` 없는 `<img>` / `<div>`·`<span>` onclick 안티패턴 / 깨진 이미지 참조 / 남은 더미·미완성 표시(`lorem ipsum`·`TODO`·`FIXME`)·깨진 글자(`�`)**. 통과(exit 0)한 뒤 아래 사람 점검으로.

## 2. 사람이 보는 self-QA (판단이 필요한 것)

기계가 못 잡는, 눈과 판단이 필요한 부분.

- [ ] **디자인 시안 대조** — 컴포넌트·간격·색·타이포가 디자인과 맞나. (틀어진 곳은 마크업에서 고칠지, 디자인 확인이 필요한지 구분)
- [ ] **반응형** — 모바일~데스크톱에서 콘텐츠가 깨지거나 가로 스크롤이 생기지 않나. 브레이크포인트 전후.
- [ ] **상태 자리** — 빈/로딩/에러 상태의 자리가 마크업에 있나(기획·디자인에 정의됐다면).
- [ ] **시맨틱·구조** — 랜드마크(`header`/`nav`/`main`/`footer`), 제목 위계가 맞나.
- [ ] **접근성** — `reference/a11y-checklist.md`의 '항상' 항목 + 페이지 성격에 맞는 '판단' 항목.
- [ ] **SEO·메타** — `reference/seo-checklist.md`. 소유 위치 존중(바깥 소유면 직접 안 씀), 값은 비워두지 않음, OG/Twitter 들어감.
- [ ] **이미지** — `reference/image-optimization.md`. webp 변환·SEO 파일명·설명형 alt·버전 스탬프.
- [ ] **키보드·대비 빠른 확인** — Tab으로 핵심 흐름이 닿나, 포커스가 보이나, 본문 대비 충분한가.
- [ ] **오탈자·카피** — 보이는 텍스트(제목·본문·버튼·라벨)를 훑어 오탈자·어색한 문장을 **플래그**한다. 단: (1) **자동 수정하지 않는다** — 카피는 보통 콘텐츠·마케팅 소유라 짚어서 확인받는다. (2) **의도적 오타·스타일라이즈는 존중** — 브랜드 슬로건(예: "smoooth"), 말장난, 상표는 오탈자가 아니다. 확실치 않으면 "이거 의도인가요?"로 묻는다. (Claude가 문맥으로 판단; 한글 맞춤법은 사전 도구보다 Claude가 정확)

## 3. 2차로 넘기는 것 (마크업 밖)

- **디자인 최종 승인** → 디자이너.
- **기능·상호작용·E2E·API 연동 후 동작** → 프론트 단계(web-frontend).
- **실기기·다양한 브라우저·정밀 스크린리더 테스트** → 프론트/전담 QA. 고-stakes 페이지일수록.

마크업은 "구조·표현·자동 점검·핵심 self-QA"까지 책임지고, 그 위는 넘긴다.

- 디자인과 틀어져 확인이 필요한 지점
