# SEO — 비례적 판단 가이드

**SEO도 이 목록을 전부 채우는 게 목표가 아니다.** 이 페이지가 검색·외부 유입 대상인지, 그게 얼마나 가치 있는지에 맞춰 *어디까지* 챙길지 판단한다. (a11y와 같은 성격 — 고정 완수가 아니라 페이지에 맞는 투자.)

**소유 분배를 먼저 기억한다:**

- **기획** — 키워드·검색 의도·URL/IA 구조. (무엇을, 어떤 주소로 노출할지)
- **마크업** — 온페이지/구조적 SEO. *이 문서가 다루는 범위.*
- **프론트** — ⚠ 크롤링 가능성(CSR)·성능(Core Web Vitals)·SPA 메타 갱신. *마크업이 아무리 완벽해도 여기서 막히면 검색에 안 잡힌다.*

마크업은 두 층으로 나뉜다(a11y와 동일):

- **항상** — 거의 공짜다. 사실 대부분 *올바른 마크업/문서 메타*와 같은 것이라, 검색을 안 노려도 해두면 손해가 없다.
- **판단** — 비용이 든다(추가 메타·구조화 데이터·검증). 검색·공유 유입이 가치 있는 페이지일수록 챙기고, 로그인 뒤 내부 화면이면 대부분 건너뛴다(오히려 `noindex`가 맞기도).

> 보너스: 아래 '항상' 항목 다수(고유 title·제목 위계·시맨틱·alt·진짜 링크·lang)는 **a11y '항상'과 그대로 겹친다.** 싼 기본기 하나가 접근성과 검색에 동시에 값을 낸다.

---

## 항상 (거의 공짜 — 페이지 무관하게)

> ⚠ **소유 먼저:** 아래 `<head>` 항목(title·description·OG·Twitter·charset·viewport·lang)은 **외부 SEO 메타 관리자**나 **공유 head include(`$변수`)** 가 소유하는 경우가 많다. 그러면 페이지에 태그를 직접 쓰지 말고 그쪽에 값만 넣는다(중복·충돌 방지). 판단 기준은 SKILL.md의 "렌더링·메타 소유 파악" 참조. — 아래 체크는 "그 값이 *어딘가에서* 채워졌는가"이지 "페이지에 태그를 박았는가"가 아니다.

- [ ] 페이지마다 **고유한 `<title>`** — 핵심을 앞에, 간결하게(~60자). 모든 페이지가 같은 제목이면 안 된다. **비워두지 말 것** — 페이지 콘텐츠·`<h1>`·사이트명으로 실제 값을 채운다.
- [ ] 페이지마다 고유한 `<meta name="description">` — 검색결과 미리보기에 쓰인다(랭킹 요인은 아니지만 클릭률에 영향). **비워두지 말 것** — 페이지 내용을 1~2문장(~150자)으로 요약해 채운다.
- [ ] `<meta name="keywords">` — 콘텐츠에서 뽑은 관련어 몇 개로 채운다. (참고: 구글 등 주요 검색엔진은 랭킹에 거의 반영하지 않는다 — '성의'·내부 참고용. 그래도 비워두기보다 채운다.)
- [ ] **Open Graph + Twitter Card (팀 필수)** — `og:title`·`og:description`·`og:image`·`og:url`·`og:type`, `twitter:card`(보통 `summary_large_image`). 공유·메신저에서 카드 미리보기. 값은 title·description과 같은 소스에서 채우고, 위 소유 규칙을 따른다(SEO 관리자/공유 head 변수/페이지).

> **메타 값 채우기:** 비워두지 말고 *페이지 콘텐츠에서 뽑은 실제 값*을 채운다(lorem ipsum 금지 — 더미라도 실제 내용 기반). 단 **소유 위치에 맞게** — 외부 SEO 관리자면 그 입력값으로, 공유 head 변수면 `$pageTitle`·`$metaDescription`에 대입, 페이지 소유면 태그에.
>
> 예 (공유 head 변수 소유 시):
> ```php
> <?php
> $pageTitle       = "제품 소개 | 회사명";
> $metaDescription = "회사명 제품 X를 한눈에. 주요 기능과 가격을 안내합니다.";
> $metaKeywords    = "제품X, 주요기능, 가격";
> ?>
> ```
- [ ] `<meta charset="utf-8">` 와 `<meta name="viewport" content="width=device-width, initial-scale=1">`.
- [ ] 페이지에 `lang` 속성.
- [ ] 제목 위계(`h1`~`h6`)가 순서대로, `h1`은 페이지 주제 하나. (크기 때문에 레벨 바꾸지 않기)
- [ ] 시맨틱 구조(`header`/`nav`/`main`/`article`/`section`) — 검색엔진이 본문·주제를 이해한다.
- [ ] 의미 있는 이미지에 내용을 담은 `alt`(이미지 검색·문맥). 장식용은 `alt=""`.
- [ ] 링크는 진짜 `<a href>` — JS `onclick`만으로 이동하면 크롤러가 못 따라간다. 앵커 텍스트는 "여기 클릭" 말고 목적지를 설명.

> 위는 어기면 "SEO를 덜 챙긴 것"이 아니라 "마크업/문서 머리를 틀리게 한 것"에 가깝다. 그래서 거의 항상 옳다.

## 판단 (페이지 성격에 따라)

검색·공유 노출이 가치 있는 페이지에서 골라 챙긴다.

- [ ] **구조화 데이터(JSON-LD, schema.org)** — Article·Product·BreadcrumbList·FAQ 등. 검색 리치결과(별점·빵부스러기·FAQ)를 노리는 공개 페이지에서 가치.
- [ ] **`<link rel="canonical">`** — 같은 콘텐츠가 여러 URL로 닿을 때 대표 주소 지정(중복 방지).
- [ ] **`<meta name="robots">`** — 의도적 `index`/`noindex` 제어. (내부·중복·임시 페이지는 `noindex`가 맞기도)
- [ ] `hreflang` — 다국어/지역별 페이지가 있을 때.
- [ ] **이미지 공간 예약(레이아웃 흔들림=CLS 방지)** — 이미지가 늦게 떠도 아래 콘텐츠가 안 밀리게 자리를 미리 잡는다. **방법은 무관**: `width`/`height` 속성, CSS `aspect-ratio`, 고정 크기 컨테이너 중 프로젝트 방식대로. (다운로드 속도 얘기가 아니다 — 그건 파일 크기·포맷·lazyload, 프론트 몫)

> `sitemap.xml`·`robots.txt`는 사이트 단위라 기획·배포 쪽에 가깝다. 마크업 단계에서 정하면 그 결정을 핸드오프에 적어 넘긴다.

## 프론트로 넘긴다 (마크업 소유 아님 — 핸드오프에 명시)

- [ ] ⚠ **크롤링 가능성** — PHP가 API만 주고 화면을 JS가 그리는 구조(CSR)면, 크롤러가 본문을 못 보거나 늦게 볼 수 있다. SSR·프리렌더·하이드레이션 여부는 **프론트/아키텍처 결정.** 검색이 중요한 공개 페이지일수록 이걸 먼저 확인하라고 넘긴다.
- [ ] **성능(Core Web Vitals: LCP/CLS/INP)** — 랭킹 요인. 프론트 소유.
- [ ] **SPA 라우팅의 동적 메타/캐노니컬 갱신** — 라우트 전환 시 `title`·메타가 바뀌게 하는 건 프론트.

## 선 긋기 (어디까지 할지)

- **이 페이지가 검색·외부 유입 대상인가?** 로그인 뒤 내부 도구·관리자 화면이면 SEO는 대부분 무의미(또는 `noindex`). 공개 랜딩·블로그·상품·문서 페이지면 끌어올린다.
- **공유(SNS) 유입을 노리나?** → Open Graph.
- **검색 리치결과를 노리나?** → 구조화 데이터.
- **검색이 중요한데 CSR 구조인가?** → 가장 먼저 프론트에 크롤링 가능성을 확인시킨다. 다른 SEO를 다 해도 여기서 막히면 헛수고.

## 핸드오프 — "범위 결정"으로 적는다

마무리하면 `docs/03-markup.md`에 SEO도 결함 목록이 아니라 **범위 결정**으로 남긴다:

- 챙긴 온페이지 요소(메타·구조화 데이터 등)
- **의도적으로 건너뛴 것 + 이유**(내부 페이지·검색 무관 등)
- **프론트로 넘기는 것** — 크롤링 가능성(CSR) 확인, 성능, 동적 메타 갱신
- 기획/배포로 넘기는 것 — sitemap·robots·URL 구조 결정(있다면)
