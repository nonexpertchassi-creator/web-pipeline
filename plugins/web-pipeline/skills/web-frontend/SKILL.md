# 웹 프론트엔드 (Frontend)

마크업에 데이터와 동작을 붙여 살아있는 화면으로 만드는 단계. 이 프로젝트는 PHP가 API만 제공하고, 프론트는 브라우저에서 도는 JavaScript가 그 API를 호출해 화면을 채운다.

## 시작 전: 이전 기록 읽기

작업 폴더에 `docs/03-markup.md`(마크업 구조·동적 영역)와 `docs/01-planning.md`(API 계약)가 있으면 **먼저 읽는다.** 어디에 데이터가 들어가고 어떤 API를 부르는지가 거기 적혀 있다.

## 시작 전 확인

- 연동할 API 계약(엔드포인트·요청/응답 형식)이 있는가. 없으면 기획/백엔드와 먼저 맞춘다.
- 프레임워크: 바닐라 JS인가, React/Vue 등인가. (지침은 공통 원칙 위주, 프레임워크별 관용에 맞게 적용)
- 인증 방식: 토큰(JWT)·세션·쿠키 중 무엇인가.

## 작업 순서

1. **API 레이어 분리** — fetch 호출을 화면 코드에 흩지 말고 한 곳(`api.js` 등)에 모은다. 베이스 URL·헤더·에러 처리를 공통화.
2. **데이터 가져와 렌더** — 화면 진입 시 필요한 데이터를 불러와 마크업에 채운다. 세 가지 상태를 항상 처리: **로딩 / 성공 / 에러**. 빈 결과(빈 배열)도 별도 처리.
3. **상태 관리** — 화면이 기억해야 할 값(목록·선택·폼 입력)을 어디에 둘지 정한다. 작은 화면은 지역 변수/컴포넌트 상태, 공유가 필요하면 한 단계 위로. 처음부터 거대한 상태 라이브러리를 깔지 않는다.
4. **이벤트·폼 처리** — 사용자 입력을 받아 API에 보낸다. 제출 중 버튼 비활성화, 중복 제출 방지, 성공/실패 피드백을 붙인다.
5. **검증** — 클라이언트 검증은 UX용 보조일 뿐, **신뢰 경계 검증은 서버(PHP)가 한다**. 클라이언트 검증만 믿고 서버 검증을 빼지 않는다.

## API 연동 핵심

- `fetch` 응답은 `res.ok`를 반드시 확인한다. 4xx/5xx도 reject되지 않으므로 직접 분기.
- 네트워크 에러(try/catch)와 HTTP 에러(`!res.ok`)를 둘 다 처리.
- 인증 토큰은 공통 헤더로 주입. 만료(401) 시 처리 흐름을 정해둔다.
- 같은 데이터를 여러 번 부르지 않게 한다 (중복 요청 가드).

## 보안·견고성 (생략 금지)

- **XSS 방지**: API 응답을 `innerHTML`로 그대로 꽂지 않는다. `textContent`를 쓰거나 프레임워크의 안전한 바인딩을 쓴다.
- **데이터 손실 방지**: 제출 실패 시 입력값을 날리지 않는다.
- **서버 검증 신뢰**: 보안·권한 판단은 서버가 한다. 프론트는 UX만.

## 원칙

- **네이티브·표준 우선.** 폼 검증은 HTML 속성(`required`, `type=email`)으로 1차 처리, 부족할 때만 JS 보강. 날짜 선택·유효성 등 브라우저가 주는 건 다시 만들지 않는다.
- **상태는 작게 시작한다.** 필요해지기 전에 전역 상태/라우터/빌드 도구를 미리 깔지 않는다.
- **로딩·에러·빈 상태는 기본값이다.** 정상 흐름만 만들면 실사용에서 반드시 깨진다.
- **API 호출과 렌더를 분리한다.** 데이터 가져오기와 화면 그리기가 섞이면 디버깅이 지옥이 된다.
