# 웹 기획 (Planning)

마크업이나 코드를 짜기 전에 "무엇을 왜 만드는지"를 먼저 확정하는 단계. 산출물은 디자인·마크업 단계의 입력이 된다.

## 시작 전 확인

다음이 불명확하면 코드/화면 설계로 넘어가지 말고 먼저 물어본다.

- **목적**: 이 화면/기능이 푸는 문제는 무엇인가. 성공 기준은?
- **사용자**: 누가, 어떤 맥락에서 쓰는가. (PC/모바일, 로그인 여부)
- **범위**: 이번에 만들 것과 안 만들 것의 경계.
- **연동**: PHP 백엔드 API와 무엇을 주고받는가. (이 프로젝트는 백엔드/프론트 분리 구조)

## 작업 순서

1. **요구사항 정리** — 기능 요구사항(무엇을 한다)과 비기능 요구사항(성능·접근성·보안)을 나눠 적는다. 각 요구사항은 검증 가능한 문장으로.
2. **정보구조(IA)** — 화면과 데이터를 계층으로 정리한다. 사이트맵(화면 트리)을 만든다.
3. **유저플로우** — 핵심 시나리오별로 사용자가 거치는 화면 흐름을 화살표로 그린다. 분기(성공/실패/빈 상태)를 빠뜨리지 않는다.
4. **화면 목록 + 와이어프레임** — 화면마다 들어갈 요소(영역, 데이터, 액션)를 글머리로 정의한다. 픽셀 디자인이 아니라 "무엇이 어디에"만.
5. **API 계약 초안** — 각 화면이 필요로 하는 데이터와 액션을 백엔드 엔드포인트 후보로 적는다. (예: `GET /api/posts`, `POST /api/posts`) 프론트/백엔드가 같은 그림을 보게 하는 게 목적.

## 산출물 형식

- 요구사항: 표 또는 번호 목록 (ID · 설명 · 우선순위)
- 사이트맵/유저플로우: 텍스트 트리 또는 mermaid 다이어그램
- 화면 정의: 화면별 섹션 + 요소 목록
- API 계약: 메서드 · 경로 · 요청/응답 필드

문서로 남길 때는 마크다운 한 파일로 정리한다. 도식이 필요하면 mermaid를 쓴다.

## 원칙

- **화면마다 빈 상태·로딩·에러를 정의한다.** 정상 흐름만 그리면 프론트에서 반드시 빠진다.
- **데이터의 출처를 명시한다.** 화면의 모든 값은 어느 API에서 오는지 적는다.
- **우선순위를 매긴다.** 전부 P0이면 기획이 아니다. MoSCoW(필수/권장/선택/제외)로 가른다.
- 기획서는 합의 도구다. 혼자 완벽하게 쓰기보다 결정이 필요한 지점을 드러내는 게 낫다.

## 핸드오프 기록 (필수)

기획이 끝나면 결과를 **반드시 파일로 남긴다.** 구두·메신저 전달은 휘발되므로, 회의에 없던 디자이너도 이 파일만 읽으면 따라잡을 수 있게 한다.

- 저장 위치: 작업 폴더의 `docs/01-planning.md`
- 양식: 아래 섹션을 포함한다.
  - 한 줄 요약 (이 화면/기능이 푸는 문제)
  - 요구사항 (필수/권장/선택)
  - 사이트맵 / 유저플로우
  - 화면별 정의 (요소·데이터·상태)
  - API 계약 초안
  - 미결 사항 (결정이 필요한 지점)
- 작성 후 사용자에게 "이 기록을 디자이너에게 공유하라"고 안내한다. 노션 등 협업 도구를 쓰면 이 마크다운을 그대로 붙여넣을 수 있다.
